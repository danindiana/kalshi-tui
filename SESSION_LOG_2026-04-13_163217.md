## Incident 1 — 2026-04-12 21:00 CDT (LIVE, $6.88 deployed)

### What happened
- BTC spot: $71,212 | Model prediction: **$75,999** | Error: +$4,787
- Scaled MAE: $894 × √20 = **$3,999** (training MAE × √mins_to_settlement)
- Gaussian assigned 77–83% win probability to all near-ATM YES strikes
- 5 LIVE orders placed, $6.88 deployed against strikes $71,250–$72,250

### Root cause: MAE inflation makes Gaussian useless at this horizon
With MAE=$3,999 and std_dev=$4,999 (MAE×1.25), the probability distribution is so wide
that nearly any bullish prediction produces 70%+ model_prob on all candidates.
The model's claimed edge was purely an artifact of:
1. A wildly bullish raw output ($75,999 — likely LSTM runaway/hallucination)
2. Massive uncertainty that, paradoxically, confirmed the bullish case for near-ATM strikes

### Example math
```
Strike=$72,250, pred=$75,999, std_dev=$4,999
z = (72250 - 75999) / 4999 = -0.750
model_prob = 1 - norm.cdf(-0.750) = 0.773 (77%)
market_price = 0.25 (25¢)
edge = 0.773 - 0.25 = +52.3%  ← large but fictional
```

## Incident 2 — 2026-04-13 00:30 CDT (LIVE, $8.46 deployed, hit daily limit)

### What happened
- BTC spot: $70,878 | Model prediction: $71,494 | Error: +$616
- Scaled MAE: $119 × √30 = **$651** (used MAE_BASE fallback — training_metrics.json may not have
  been present at run time, or was different)
- 6 LIVE orders placed at strikes $70,900–$71,400, total $8.46
- Contracts priced at **2–6¢** — market assessed 2–6% probability; model claimed 55–70%

### Root cause: Cheap contract pricing inflates edge calculation
The edge formula is simply `model_prob - market_price`. When market price is 2¢:
```
Strike=$71,300, pred=$71,494, MAE=$651, std_dev=$814
z = (71300 - 71494) / 814 = -0.238
model_prob = 1 - norm.cdf(-0.238) = 0.594 (59%)
market_price = 0.02 (2¢)
edge = 0.594 - 0.02 = +57.4%   ← 29.7× ratio — market says 2%, model says 59%
```
A 2¢ contract price represents strong market consensus that the strike won't hit.
The Gaussian edge formula has no mechanism to discount claims this extreme.

### Outcome
All 6 contracts likely expired worthless. BTC was trending down toward $70,878 at 00:30;
the contracts required BTC > $70,900–$71,400 at 01:00 settlement.
Daily deployment cap hit ($8.46 ≥ $8.00 limit), blocking all subsequent runs on 04-13.

## Bugs Found

### Bug 1: No MAE ceiling before edge evaluation
**File:** `auto_trader.py`  
**Impact:** High-MAE runs generate phantom edges on every near-ATM candidate.  
**Root:** `mae = mae_base * sqrt(mins)` has no upper bound. At 20–30 min horizon with
a $894 trained MAE, uncertainty = $4,000–$5,000 (5–7% of BTC price).

### Bug 2: No minimum contract price filter
**File:** `auto_trader.py`  
**Impact:** Contracts at 2–6¢ appear to have enormous edges (30×+ model-to-market ratio).  
**Root:** Market prices this low represent near-certain consensus the strike won't hit;
the edge formula treats them as free money.

### Bug 3: No model-to-market probability ratio cap
**File:** `auto_trader.py`  
**Impact:** Belt-and-suspenders gap — even contracts above 10¢ can have unrealistic ratios
if prediction is far from current price.

### Bug 4 (minor): `daily_loss()` docstring misleading
**File:** `auto_trader.py`  
**Impact:** Comment said "trades that failed" but tracks all placed trades. Renamed conceptually
to "daily deployment cap" with corrected docstring.

## Fixes Applied (2026-04-13T16:32 CDT)

### 1. MAE ceiling gate — `auto_trader.py`
```python
MAE_MAX_DOLLARS = float(os.environ.get("MAE_MAX_DOLLARS", "2000"))
```
Added immediately after model inference. If `mae_current > MAE_MAX_DOLLARS`, logs the run
with 0 opportunities and returns. Service default: `MAE_MAX_DOLLARS=2000`.

**Would have caught:** Incident 1 (MAE=$3,999 > $2,000 → skip)

### 2. Minimum contract price gate — `auto_trader.py`
```python
MIN_CONTRACT_CENTS = float(os.environ.get("MIN_CONTRACT_CENTS", "10"))
```
Added inside the opportunity evaluation loop. Rejects any contract where
`market_price * 100 < MIN_CONTRACT_CENTS`. Service default: `MIN_CONTRACT_CENTS=10`.

**Would have caught:** Incident 2 contracts at 2¢, 3¢, 6¢

### 3. Maximum model-to-market probability ratio gate — `auto_trader.py`
```python
MAX_PROB_RATIO = float(os.environ.get("MAX_PROB_RATIO", "3.0"))
```
Added inside the opportunity evaluation loop. Rejects any contract where
`model_prob / market_price > MAX_PROB_RATIO`. Service default: `MAX_PROB_RATIO=3.0`.

**Would have caught:** Incident 2 contracts at 16¢ (ratio 4.6× > 3.0×).
Note: only 70900@44¢ survives both gates — ratio 1.7×, 44¢ price, edge +32.7% (legitimate).

### 4. `daily_loss()` docstring fix — `auto_trader.py`
Updated to accurately describe it as a deployment cap, not a realized-loss tracker.

### 5. Service file updated — `/etc/systemd/system/kalshi-auto-trader.service`
Added three new `Environment=` lines for `MAE_MAX_DOLLARS`, `MIN_CONTRACT_CENTS`, `MAX_PROB_RATIO`.

## Open Issues / Next Steps

### High priority
- **LSTM is predicting $75,999 for a $71K market** — this is a runaway prediction.
  The model may have diverged during retraining. Consider:
  - Adding a model sanity check: if `|prediction - current_price| > X%`, treat as unreliable
  - Retraining with better regularization or reducing seq_len
  - Logging the raw model output (before scaler inverse) to diagnose

- **`daily_loss()` tracks deployment cost, not P&L** — there is no outcome tracking.
  Contracts placed at 00:30 almost certainly expired worthless ($8.46 lost),
  but the DB has no mechanism to mark them. See v0.4 ideas in STATUS.md.

### Medium priority
- **Model type is "absolute" predicting end-of-day price** — using it for 20-30 min
  windows with sqrt(T) MAE scaling is theoretically incorrect. The absolute-price LSTM
  predicts the 5PM settlement, not "price in T minutes." The sqrt(T) band is a
  random-walk approximation that massively inflates uncertainty.
  Better: retrain as a proper short-horizon delta model.

- **`MIN_CONTRACT_CENTS=10` may be too permissive** — even 10–15¢ contracts can be
  thinly traded. Consider 15¢ if excessive false edges persist.

### Low priority
- Outcome auto-tracking: poll Kalshi settled markets and update trade P&L in DB
- Daily P&L summary notification
- Retrain LSTM weekly on fresh Gemini candles
