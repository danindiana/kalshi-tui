# Session Log — 2026-04-13_083437

## Goal

Replace the systemd timer + service with a human-operated, single-entry-point
loop that the operator starts on demand and runs until reboot or manual stop.

---

## Motivation

The systemd timer was always-on and caused the broken LSTM model to keep trading
overnight after the daily loss limit reset at midnight, compounding losses.
The new model wants human oversight: operator runs the wizard, reviews the plan,
approves execution, and the loop stays up as long as they choose.

## What Was Built

### 1. `wizard.py` — new `--loop` and `--interval` flags

Two new args added to argparse:

| Flag | Default | Meaning |
|------|---------|---------|
| `--loop` | off | After operator approves, run `auto_trader.py` every `--interval` minutes until Ctrl+C |
| `--interval MIN` | 30 | Minutes between auto-trader cycles |

New function `run_loop(dry_run, interval)` added in `§ 6`:
- Wraps `run_auto_trader()` in a `while True` loop
- Shows a live `MM:SS` countdown between cycles via `sys.stdout.write(\r…)`
- Catches `KeyboardInterrupt` cleanly — prints a final rule and exits
- Errors inside a single `auto_trader` run are caught and logged; the loop
  continues to the next cycle (no hard crash)

Decision block in `main()` updated:
```
skip   → wizard exits (as before)
dry    → run_loop(dry_run=True, ...)  if --loop else run once
live   → run_loop(dry_run=False, ...) if --loop else run once
```

### 2. `start_trader.sh` — tmux session manager

Single entry-point shell script. Installed at `/usr/local/bin/start_trader`.

**Flow:**

```
start_trader [extra wizard args]
  │
  ├─ Is tmux session 'kalshi' already running?
  │     YES → prompt: [a] Attach  [k] Kill & restart  [q] Quit
  │     NO  → fall through
  │
  └─ Launch new tmux session 'kalshi':
       bash -c "wizard.py --loop [extra args]"
       + keepalive "Press Enter to close" after wizard exits
     → Attach to session
```

**Usage:**
```bash
start_trader                    # 30-min interval, default args
start_trader --interval 15      # 15-min interval
start_trader --brief            # + deepseek strategy review at startup
start_trader --no-ai            # skip Ollama briefing
start_trader --interval 5 --no-ai  # fast cycling without AI (dev/test)
```

**tmux session controls:**
| Action | Keys / Command |
|--------|---------------|
| Detach (leave running) | `Ctrl+B D` |
| Re-attach | `tmux attach -t kalshi` |
| Stop the loop | `Ctrl+C` inside session |
| Kill session from outside | `tmux kill-session -t kalshi` |

## Bug Fixed: tmux command word-splitting

**Symptom:** `start_trader --brief` showed `[exited]` immediately on attach.

**Root cause:** Original script used:
```bash
tmux new-session -d -s kalshi -x 200 -y 50 bash -c "$INNER_CMD"
```
tmux receives the multiple positional args (`bash`, `-c`, `<INNER_CMD>`), joins
them with spaces, then passes to `${SHELL} -c "bash -c <INNER_CMD>"`. The outer
shell word-splits `<INNER_CMD>` so bash's `-c` only receives `cd` as its script.
Wizard never ran.

**Fix:** Pass the entire command as ONE double-quoted string:
```bash
LOOP_CMD="cd '${TRADER_DIR}' && '${PYTHON}' '${WIZARD}' --loop $*"
tmux new-session -d -s "$SESSION" -x 200 -y 50 "$LOOP_CMD"
```
tmux receives one shell-script string and runs it via `${SHELL} -c "..."` correctly.

Also added `remain-on-exit on` so the pane stays visible (showing last output)
after the wizard exits, instead of immediately closing.
