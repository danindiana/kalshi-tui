### `src/db.rs` additions

**`try_open_trader_db()`** — opens `~/.local/share/kalshi-tui/auto_trader.db` if it
exists; returns `None` if the file is absent (auto-trader never run).

**`load_auto_trades(conn)`** — SQL query across base tables (avoids joining a view on
itself), computing outcome and P&L inline:

```sql
SELECT t.id, t.placed_at, t.ticker, t.strike, t.side, t.count,
       t.price_cents, t.cost_dollars, t.edge,
       CASE
         WHEN s.market_result IS NULL THEN 'open'
         WHEN (t.side='YES' AND s.market_result='yes') OR
              (t.side='NO'  AND s.market_result='no') THEN 'win'
         ELSE 'loss'
       END AS outcome,
       CASE
         WHEN s.market_result IS NULL THEN NULL
         WHEN (t.side='YES' AND s.market_result='yes') OR
              (t.side='NO'  AND s.market_result='no')
              THEN (CAST(t.count AS REAL)) - t.cost_dollars
         ELSE -t.cost_dollars
       END AS pnl,
       r.dry_run
FROM trades t
JOIN runs r ON r.id = t.run_id
LEFT JOIN settlements_raw s ON s.ticker = t.ticker
WHERE t.status = 'placed'
ORDER BY t.id DESC
LIMIT 500
```

Reads newest-first, capped at 500 rows. SQLite WAL mode handles concurrent access
with the Python auto-trader without blocking.

## Key design decisions

**1. Read directly from base tables, not the `trade_outcomes` view**

`auto_trader.db` already has a `trade_outcomes` VIEW that joins `trades`, `runs`, and
`settlements_raw`. However, joining that view with `runs` again (to get `dry_run`) would
be a join-on-view that's less clear. The direct query avoids this and makes the P&L
computation explicit.

**2. `trader_db` is stored in `App` as `Option<Connection>`**

WAL mode allows simultaneous readers and writers without blocking. Storing the connection
avoids re-opening the file on every reload. `reload_auto_trades()` re-queries through the
same handle.

**3. Reload on `'r'`**

Auto-trade reload is wired into `request_refresh()` rather than a new keybinding.
The SQLite read is synchronous and fast (~1 ms for 500 rows), so it happens before
the async market fetch is spawned — no additional threading required.

**4. Newest-first, LIMIT 500**

Recent trades are most operationally relevant. 500 rows is enough history for any
reasonable review session without loading the full history (616+ rows at time of
writing).
