## Implemented

### `scripts/paper_trader.py` (new)
Standalone simulation backend. Reuses `run_classifier`, `check_trading_window`, and all
auto_trader constants by importing `auto_trader as at` via sys.path (safe: `run()` is guarded
by `if __name__ == "__main__"`). Runs N cycles back-to-back, places virtual bets tracked in
new DB tables, resolves settlement via live Kalshi `/markets/{ticker}` result field, and
generates a session markdown doc.

Key behaviors:
- Sizing mirrors live ratchet (MAX_STAKE_PCT, MAX_RUN_PCT, etc.) applied to the paper bankroll
- Settlement resolution: `yes`/`no` result → win/loss pnl; unsettled → mark-to-market from current ask
- Final line of stdout is `JSON:{...}` for the TUI wizard to parse
- Session doc written to `SESSION_{timestamp}.md` in workspace root

### `src/wizard.rs` (new)
Ratatui TUI wizard with 8-state machine:
`Welcome → Balance → Cycles → MaxTrades → Confirm → Running → Results / Errored`

- Centered modal layout (60% × 75%), rounded borders, dark background
- Input fields with live cursor, validation errors shown inline, Esc navigates back
- Running screen: streams subprocess stdout/stderr line-by-line with color coding
  (`[PAPER]` green, `[OK]`/`[...]` cyan, `[FAIL]`/`[STOP]`/`[ERR]` red, `[SKIP]`/`[WARN]` yellow)
- Auto-scrolls log to bottom; j/k manual scroll
- `JSON:` prefix on last subprocess line parsed into `SessionResult` → Results screen
- subprocess spawned via `tokio::process::Command` with piped stdout+stderr
- Results screen: color-coded P&L (green/red), win/loss/pending counts, session doc name

### `src/main.rs` (modified)
Added `--paper-trade` bool flag to `Cli`. If set, calls `wizard::run_wizard()` and returns
before any of the normal TUI setup. The wizard handles its own terminal teardown.

### DB additions
Two new tables in `~/.local/share/kalshi-tui/auto_trader.db`:
- `paper_sessions` — one row per wizard session (config + final P&L)
- `paper_trades` — per-trade records with settlement status and pnl
